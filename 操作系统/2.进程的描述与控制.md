# 一、前趋图和程序执行
## 1、前趋图（DAG）
- **定义**：指一个有向无循环图，它用于描述进程之间执行的先后顺序
- **结点**：指一个进程或程序段，乃至一条语句
- **有向边**：指两个结点之间存在的**偏序**或**前趋关系**
	进程 P<sub>i</sub> 和 P<sub>j</sub> 存在前趋关系表示为：（ P<sub>i</sub> ，P<sub>j</sub> ） ∈ →  或 P<sub>i</sub> → P<sub>j</sub> 
	概念：
	>初始结点：没有前趋的结点
	>终止结点：没有后继的结点
	>重量（权重）：表示该结点所含有的程序量或程序的执行时间
	>例如：P<sub>i</sub>  是 P<sub>j</sub> 的直接前趋，P<sub>j</sub> 是 P<sub>i</sub> 的直接后继
## 2、程序的顺序执行
- **定义**：仅当前一操作（程序段）执行完后，才能执行后继操作，需要按照某种先后次序执行
### （1）执行时的特征
- **顺序性**：严格按照程序所规定的次序执行
- **封闭性**：程序在封闭环境下运行，系统中所有资源的状态只有本程序才能改变它
- **可再现性**：只要初始条件相同，无论怎样执行，其结果都是相同的
## 3、程序的并发执行
### 特征
- **间断性**：并发执行的实体之间相互制约，造成程序的执行出现间断，而不连续
- **非封闭性**：多个程序共享系统资源，因而其状态由多个程序改变，从而失去封闭性
- **不可再现性**：封闭性的失去必然导致不可再现性
# 二、进程
## 1、定义
- 由于程序的并发执行，产生了一些新的特征，原有程序的概念不能充分描述并发执行问题，于是产生了进程的概念
- 进程实体（进程映像/进程）：由程序段、相关的数据段、PCB构成
- **进程是操作系统中最重要的概念**
### 典型定义
>	进程是程序的一次执行
>	进程是一个程序及其数据在处理机上顺序执行时所发生的活动
>	进程是程序在一个数据集合上运行的过程，它是系统进行资源分配和调度的一个独立单位
### 传统OS中的进程定义：
- **进程是进程实体的运行过程，是系统进行资源分配和调度的一个独立单位**
### 特征
- 结构性：由**程序段**、**相关数据段**和[进程控制块（PCB）](#（2）进程控制块（PCB）)构成
- 动态性：有“生命期” ，并发执行过程中多种状态之间转换
- 并发性：引入进程的目的
- 独立性：独立运行、独立获得资源、独立调度
- 异步性：按照各自的不可预知的速度向前推进
### 程序与进程的对比

| 程序                             | 进程                                |
| ------------------------------ | --------------------------------- |
| **静态的**：有序代码的集合，是永久的，对应着文件、可复制 | **动态的**：程序的执行，有生命期、临时的，是一个状态变化的过程 |
| 组成：代码集合                        | 组成：程序、数据和PCB                      |
| 不支持并发<br>                      | 支持并发                              |
| 不能作为独立单位参与运行                   | 具有独立性                             |
| 通过多次执行，**一个程序可对应多个进程**<br>     | 通过调用，**一个进程可包括多个程序**              |

## 2、进程管理中的数据结构
### （1）用于管理控制的数据结构
- **资源信息表（进程信息表）**：在计算机系统中，对于每个资源和每个进程都设置了一个数据结构，用于表征其实体
>包含了资源或进程的标识、描述、状态等信息以及一批指针
>通过这些指针，可以将同类资源或进程的信息表，或者同一进程所占用的资源信息表分类链接成不同的队列，便于操作系统进行查找
![](attachments/Pasted%20image%2020260331205219.png)
### （2）进程控制块（PCB）
- **定义**：为了便于系统描述和管理进程，专门定义的数据结构
>是进程实体的一部分，记录了系统所需的用于描述进程当前情况以及管理进程运行的全部信息
>**是操作系统最重要的记录型数据结构**
- **作用**：
>1）作为独立运行基本单位的标志，是进程存在于系统的唯一标志
>2）能实现间断性运行方式，支持并发执行（处理机状态信息）
>3）提供进程管理所需要的信息
>4）提供进程调度所需要的信息
>5）实现与其它进程的同步与通信
### （3）进程控制块中的信息
####        进程标识符
>		用于唯一地标识一个进程。一个进程通常有两种标识符：  内部标识符（方便系统对进程使用），外部标识符（方便用户或进程对进程的访问）
####       处理机状态
>		主要是由处理机的各种寄存器中的内容组成的：通用寄存器，指令寄存器，程序状态字，用户栈指针
####       进程调度信息
>		在PCB中还存放一些与进程调度和进程对换有关的信息，包括：进程状态；进程优先级；进程调度所需的其它信息（与采用的进程调度算法有关）；事件（阻塞原因）
####       进程控制信息
>		包括：程序和数据的地址；进程同步和通信机制； 资源清单(全部、已分配）；链接指针。
### （4）进程控制块的组织方式
- **线性方式**：实现简单开销小，查找困难，适合进程数目不多的系统
- **链接方式**：即把有相同状态进程的 PCB 分别通过链接字连接成一个队列
- **索引方式**
# 三、进程控制
## 1、OS内核
### （1）支撑功能
- **中断处理**：内核最基本的功能，是整个操作系统赖以活动的基础
- **时钟管理**：内核的一项基本功能，在OS中的许多活动都需要得到它的支撑
* **原语操作**：由若干条指令组成的，用于完成一定功能的一个过程。
>原语操作与一般过程的区别在于：它们是**原子操作**。一个操作中的所有动作要**么全做，要么全不做**。是**一个不可分割的基本单位**，在**执行过程不允许被中断**。在系统态下执行，常驻内存
### （2）资源管理功能
- **进程管理**：
>进程的调度与分派、进程的创建与撤消等；进程同步的原语、常用的进程通信原语
- **存储器管理**：
>实现将用户空间的逻辑地址变换为内存空间的物理地址的地址转换机构、内存分配与回收功能、内存保护和对换功能
- **设备管理**：由于设备管理与硬件紧密相关，因此其中很大部分也都设置在内核中
>如各类设备的驱动程序、用于缓和CPU与I/O速度不匹配矛盾的缓冲管理、用于实现设备分配和设备独立性功能的模块等
## 2、进程的创建
### （1）进程的层次结构
>在OS中，允许一个进程创建另一个进程，通常把创建进程的进程称为父进程，而把被创建的进程称为子进程。子进程可继续创建更多的孙进程，由此便形成了一个进程的层次结构。（类似于树）
- 子进程继承父进程资源，子进程随着父进程的撤销而撤销
### （2）引起创建进程的事件
	用户登录，作业调度，提供服务，应用请求
### （3）进程的创建过程
	1）申请空白PCB
	2）为新进程分配资源（内存、文件、设备、CPU时间）
	3）初始化PCB
	4）将新进程插入就绪队列，如果进程就绪队列能够接纳新进程， 便将新进程插入就绪队列
>以上操作是：OS调用进程创建语句 Creat 去操作创建的
## 3、进程的终止
### （1）引起进程终止的事件
	1）正常结束：进程的任务已经完成，准备退出运行
	2）异常结束：指进程在运行时发生了某种异常事件，使程序无法继续执行。eg:越界错误；保护错；非法指令；特权指令错；运行超时；等待超时；算术运算错；I/O故障
	3）外界干预：指进程应外界的请求而终止运行。eg:操作员或操作系统干预；父进程请求；父进程终止
### （2）进程的终止过程
	1）根据终止进程的标识符，从PCB队列中读出该进程
	2）如果终止进程还在运行就终止进程执行，将处理机分配给其它进程
	3）终止其子孙进程，防止其成为不可控进程
	4）回收所占所有资源给父进程或者系统
	5）从队列中删除PCB
## 4、进程的阻塞与唤醒
### （1）引起进程阻塞的事件
- 向系统请求共享资源失败
- 等待某种操作完成
- 新数据尚未到达
- 等待新任务到达
### （2）阻塞过程 block（主动行为）
	1）停止进程运行，让出CPU
	2）将PCB插入阻塞队列
### （3）唤醒过程 wakeup（被动行为）
	1）从阻塞队列中移出该进程
	2）将PCB状态更改为就绪
>注：当阻塞进程所期待的事件发生时，则由其它进程唤醒该进程。所谓“阻塞自己、唤醒别人”。阻塞的 block 原语和唤醒的 wakeup 原语必须成对使用，否则阻塞的进程将永久处于阻塞状态
## 5、进程的挂起与激活
### （1）挂起过程（suspend）
	1）检查被挂起进程的状态
	2）更改其状态
### （2）激活过程（active）
	1）检查被激活进程的状态
	2）更改其状态

# 四、 进程同步
## 1、基本概念
- 原因：**多道程序并发执行**在有效改善资源利用率和提高系统吞吐量的同时，也**使系统变得更加复杂**，若不实施妥善的管理，必然会因为进程对**系统资源的无序争夺**造成系统混乱及处理**结果的不确定性和不可再现性**
- **主要任务**：对多个相关进程在**执行次序上进行协调**，使并发执行的诸进程之间能按照一定的规则**有效的共享资源和相互合作**，从而使程序执行**具有可再现性**
### （1）两种制约关系
>在多道程序环境下，对于同处于一个系统中的多个进程， 由于它们共享系统中的资源，或为完成某一任务而相互合作，它们之间可能存在着以下两种形式的制约关系
- 直接制约关系：源于进程之间的相互合作
- 间接制约关系：程序并发执行时由于共享系统资源，使并发的程序见相互制约
### （2）临界资源
- **定义**：OS中把一段时间内只允许一个进程访问的资源
>系统中有各种资源，由于它们各自不同的固有特征，多个进程在共享使用它们时，如果不作特殊处理，则会出现许多意想不到甚至是完全错误的结果
- **典型案例**：生产者—消费者问题
### （3）临界区
- **定义**：进程中访问临界资源的那段代码
>显然若能保证诸进程互斥地进入自己的临界区，便可实现诸进程对临界资源的互斥访问
- 可以把一个访问临界资源的循环进程描述如下：
``` C++
while(1)
{
	entry section        //入口部分
	critical section;    //临界区部分
	exit section         //出口部分
	remainder section;   //剩余部分
}
```
### （4）同步机制应遵循的规则
- **空闲让进**：临界资源空闲时，允许进程立即进入
- **忙则等待**：资源被占用时，其他进程必须等待
- 有限等待：等待进程应在有限时间内获得资源，避免“饥饿”
- 让权等待：等待时释放 CPU，避免忙等
## 2、信号量机制
### （1）整型信号量（即一个整型量）
- 除初始化外，仅能通过两个标准的**原子操作**：**wait(S)** 和 **signal(S)** 访问，简称P、V操作
>因为 wait 和 signal 是原子操作所以他们必须同时出现
- wait 和 signal 操作可描述为：
``` C++
wait(S)
{
	while(S <= 0);    //循环体为空
	S = S - 1;
}
signal(S)
{
	S = S + 1;
}
```
### （2）记录型信号量（semaphore）
>在整型信号量机制中，在 S ≤ 0 时，需要不断测试，处于**忙等**状态，因而违背了“让权等待”的准则。在采取“让权等待”的策略后，会出现多个进程等待访问同一临界资源的情况 
- 信号量机制中除了需要一个表示资源数目的变量外，还应增加一个指针链接这些等待进程。故对原有的信号量作以下定义：
``` C++
typedef struct{
	int value;
	struct process_control_block *list;
}semaphore;
```
- 对于记录型信号量机制，wait 和 signal 原语就可以描述为：
``` C++
wait(semaphore *S)
{
	S->value -- ;
	if(S->value < 0) block(S->list);    //block(阻塞):系统原语
}
signal(semaphore *S)
{
	S->value ++ ;
	if(S->value <= 0) wakeup(S->list);  //wakeup(唤醒):系统原语
}
```
- value 信号量的物理意义：
>	资源信号量初值为资源数量；互斥信号量初值为1
>	**正值代表同类临界资源的数目**
>	**负值时其绝对值代表等待进入临界区的进程数目**
### （3）AND型信号量
- 上述进程互斥问题，是针对进程之间互斥共享一个临界资源而言，但有些应用场合中，一个进程需要获得两个或更多的资源后，才能继续执行。此时如果原语操作的执行顺序不好，进程将会进入一个**僵持状态**无法解脱（即**死锁状态**）究其原因在于资源的非完备分配
- **核心思想**：要么一次性获得所有所需资源，要么一个都不拿，进入等待
>它通过原子操作 `Swait(S1, S2, ..., Sn)` 实现：
>- 如果所有信号量 `Si ≥ 1`，则全部减 1，进程继续执行；
>- 否则，进程被阻塞，并不占用任何已检查过的资源。
>释放时使用 `Ssignal(S1, S2, ..., Sn)`，将所有信号量加 1，并唤醒等待者。
``` C++
Swait(S1, S2, ..., Sn)
{
	while(TRUE)
	{
		if(S1 >= 1 && ... && Sn >= 1)
		{
			for(i = 1; i <= n; i++) Si--;
			break;
		} 
		else 
		{
			将该进程放入与首个满足 Si < 1 条件的 Si 相关的等待队列中，
			并将此程序的程序计数器设置为 Swait 操作的起始位置
		}
	}
}
Ssignal(S1, S2, ..., Sn)
{
	while(TRUE)
	{
		for(i = 1; i <= n; i++)
		{
			Si++;
			将与 Si 相关的队列中的所有等待进程移入就绪队列中
		}
	}
}
```

| 问题    | 传统信号量       | AND型信号量   |
| ----- | ----------- | --------- |
| 死锁风险  | 高（因申请顺序不一致） | 极低（原子性获取） |
| 资源碎片  | 可能持有部分资源空等  | 不会持有部分资源  |
| 执行确定性 | 结果依赖调度顺序    | 行为更可预测    |

### （4）一般信号量集
- **定义**：一种对基本信号量机制的扩展，用于更灵活、高效地处理多个资源的批量申请与释放
#### 两个原子操作
- 1） Swait(S₁, t₁, d₁, ..., Sₙ, tₙ, dₙ)
	**含义**：尝试同时对多个信号量进行检查和分配。
	 **参数解释**：
    - `Sᵢ`：第 i 个信号量；
    - `tᵢ`：该信号量的**下限值**（threshold），即至少要有多少资源才允许分配；
    - `dᵢ`：本次申请的**资源数量**（demand）。
- 2）Ssignal(S₁, d₁, ..., Sₙ, dₙ)
	**含义**：一次性释放多个资源。
	- `注意`：整个 Swait 操作是**原子的**——要么全部成功，要么全部失败并阻塞，避免部分分配导致死锁或资源浪费
``` C++
Swait(S1, t1, d1, …, Sn, tn, dn)  
{
	while(1)
	{
	   if ( (S1 >= t1) && … &&( Sn >= tn ) ) 
	    {
			for (i = 1; i <= n; i++)  Si = Si - di;  
			break;
		}
		else 
		{
		    将正在执行的进程置于第一个满足 Si < ti 条件的 Si 的等待队列中，
			并将其程序计数器设置为 Swait 操作的起始位置。
		}
	}
}
signal(S1, d1, …, Sn, dn)
{
	while(1)
	{
		for (i = 1; i <= n; i++)
		{
			Si = Si + di;
			将与 Si 相关的队列中所有等待中的进程移至就绪队列。
		}
	}
}
```
#### 与AND信号量的关系
- **AND 型信号量** 是信号量集的特例：
    - 所有 `tᵢ = 1`，`dᵢ = 1`；
    - 即要求每个资源至少有 1 个，且各申请 1 个。
- **信号量集更通用**：支持任意阈值和需求量，灵活性更强。
#### 几种特殊情况
- Swait(S, t, d)
>	此时在信号量集中只有一个信号量 S， 它本次申请 d 个资源，当现有资源数少于 t 时，不予分配
- Swait(S, 1, 1)
>	此时的信号量集已蜕化为一般的记录型信号量(S＞1时)或互斥信号量(S =1时)
- Swait(S, 1, 0)
>	这是一种很特殊且很有用的信号量操作。当S≥1时，允许多个进程进入某特定区；
>	当S变为0后，将阻止任何进程进入特定区。换言之，它相当于一个可控开关

- 进餐完毕，放下筷子继续思考

## 3、读者—写者问题
### （1）问题描述
一个数据文件或记录，可被被多个进程共享，我们把只要求读该文件的进程称为"读者进程"，其他进程称为"写者进程"。
**读者-写者问题的规则：**
- 允许多个读者进程同时读一个共享文件，因为读操作不会使数据文件混乱
- 不允许两个或两个以上写者进程同时访问共享文件，因为这种访问将会引起混乱
- 不允许一个写者进程和其他读者进程同时访问共享文件，因为这种访问将会引起混乱
所谓"读者-写者问题"——是指保证一个Writer进程必须与其它进程互斥地访问共享对象的同步问题。
### （2）利用记录型信号量解决读者-写者问题
- 为实现 Reader 进程和 Writer 进程间的互斥，设置一个互斥信号量 Wmutex，其初值为 1；
- 设置一个整型变量 Readcount ，记录正在读的读者进程数。其初值为 0 ；
- 由于只要有一个 Reader 进程在读，便不允许 Writer 进程去写，因此第一个读者进程需要执行 wait（Wmutex）操作，即当 Readcount = 0 时，Reader 进程才需要执行 wait（Wmutex）操作。
- **算法描述**：
``` c++
semaphore Rmutex=1，Wmutex=1;
int Readcount= 0;
void Reader ()
{
  do{   
       wait(Rmutex);
       if (Readcount==0) wait(Wmutex);
       Readcount = Readcount + 1;
       signal(Rmutex);
       Reading;
       wait(Rmutex);
       Readcount = Readcount – 1;
       if (Readcount==0) signal(Wmutex);
       signal(Rmutex);
	}while(TRUE);
}
void Writer ()
{ 
	do{   
        wait(Wmutex);
        Writing;
        signal(Wmutex);
    }while(TRUE);
}
```
**【分析】**
- 当第一个读者在读文件时，后续读者也可进入临界区读该文件，后续写者不能写（在Wmutex 上阻塞）；待所有读者退出时，由最后退出的读者唤醒一个写者。
- 当有一个写者在写时，后续写者不能写，在 Wmutex 上阻塞；后续读者不能读，其中第一个读者在 Wmutex 上阻塞，其余读者在 Rmutex 上阻塞。该写者退出时，唤醒一个写者或读者。
- **问题**：上述算法实际上是“优先读者"算法，当有读者正在读，且后续读者源源不断到来时，写者将长期得不到服务，即写者会被"饿死"。
### （3）读写公平算法
为解决写者饿死问题，可以考虑"优先写者"的算法：
1. 写者与读者公平竞争
2. 写者相对于读者优先
在一个写者到达时如果有正在读的读者，那么该写者只要等待读者完成，而不必等候其后续到来的读者就可以进行写操作。
- 排队信号量Order：用于保证写进程封锁其后续到达的读进程的信号量
``` c++
Semaphore rcmutex=1, wmutex=1, order=1; 
int readcount=0;
void reader(){
    do{
        wait(order);
        wait(rcmutex);
        if( readcount==0) wait(wmutex);
        readcount=readcount+1;
        signal(rcmutex);
        signal(order);
        reading
        wait(rcmutex);
        readcount:=readcount-1;
        if (readcount==0) signal(wmutex);
        signal(rcmutex);
    }while(TRUE);
}
void writer(){
    do{
        wait(order);
        wait(wmutex);
        writing
        signal(wmutex);
        signal(order);
    }while(TRUE);
}
```
### （4）写者优先算法
保证写者相对于读者优先，需要提高写者进程的优先级。除增加一个排队信号量 order ，让写者之后的读者都在此信号量排队外，同时增加记录写者数量的变量 writercount 和互斥访问该变量的信号量 wcmutex
- 设置4个互斥信号量和2个共享计数变量:
	- 互斥信号量 order：第一个写进程执行 wait(order) 操作，用于封锁后续读进程。最后一个写进程执行 signal(order) 操作，使后续读进程得到排队的权利。
	- 整型变量 Readcount：初值为 0，用于读进程计数。
	- 互斥信号量 rcmutex：用于读进程互斥访问共享变量 rcount。
	- 互斥信号量 wmutex：第一个读进程执行 wait(wmutex)用于封锁写进程。
	- 整型变量 writecount：初值为 0，用于写进程计数。
	- 互斥信号量 wcmutex：用于写进程互斥访问共享变量 writecount
``` c++
// 初始化
semaphore rcmutex=1,wcmutex=1,wmutex=1,order=1; 
int readcount=0,writercount=0;
void reader(){
    do{
        wait(order);
        wait(rcmutex);
        if (readcount==0) wait(wmutex);
        readcount=readcount+1;
        signal(rcmutex);
        signal(order);
        reading
        wait(rcmutex);
        readcount=readcount-1;
        if (readcount==0) signal(wmutex);
        signal(rcmutex);
    }while(TRUE);
}
void writer(){
    do{
        wait(wcmutex);
        writecount=writecount+1;
        if (writecount==1) wait(order);
        signal(wcmutex);
        wait(wmutex);
        writing;
        signal(wmutex);
        wait(wcmutex);
        writecount=writecount-1;
        if (writecount==0) signal(order);
        signal(wcmutex);
    }while(TRUE);
}
```
# 六、管程机制

## 1、管程的引入
信号量机制方便、有效，但每个要访问临界资源的进程都必须自备同步操作wait(S)和signal(S)，使大量的同步操作分散在各个进程中，给系统管理带来麻烦；会因同步操作使用不当而导致系统死锁。
- **管程的基本思路是：** 把分散在各进程中的临界区集中起来进行管理，并把系统中的共享资源用数据结构抽象地表示出来。
## 2、管程的定义
- **定义**：当共享资源用数据结构表示时，资源管理程序可用对该数据结构进行操作的一组过程来表示，如资源的请求和释放过程 request 和 release 。我们把这样一组相关的数据结构和过程一并称为管程
**管程由四部分组成：**
1. 为管程赋予一个名字
2. 局部于管程的共享变量说明
3. 对该数据结构进行操作的一组过程
4. 对局部于管程的数据设置初值的语句
![](attachments/Pasted%20image%2020260418175916.png)
## 3、管程的特性
- 局部于管程的数据结构，仅能被局部于管程的过程访问，任何管程外的过程都不能访问它
- 局部于管程的过程仅能访问管程内的数据结构
- 管程把共享变量和对它的操作的若干过程 **封装** 起来，所有进程要访问临界资源时，都必须经过管程才能进入，而管程**每次只准许一个**进程进入管程，从而实现了进程互斥
### 1、条件变量
- 利用管程实现进程同步时，必须设置两个同步操作原语 wait 和 signal。
- 某进程通过管程请求临界资源未能满足时，管程调用 wait 原语将该进程阻塞，并把它排到等待队列上。
- 当另一进程访问完成并释放该资源后，管程调用 signal 原语唤醒等待队列中的队首进程
等待的原因可有多个，为区别它们，引入了**条件变量condition**。
``` c++
condition x, y;  // 条件变量的说明形式
x.wait;         // 对应的原语调用形式
x.signal;
```
- **说明**：
	- x.signal 的操作与信号量中的 signal 操作不同
	- x.signal 操作的作用是重新启动一个被阻塞的进程，但如果没有被阻塞的进程，则x.signal 操作不产生任何后果
	- signal 总是要执行 s = s+1操作，因而总是要改变信号量的状态
### （2）利用管程解决生产者-消费者问题
包括两个过程：
- put(item) 过程：生产者利用该过程将自己生产的产品投放到缓冲池中，并用整型变量count 来表示在缓冲池中已有的产品数目，当 count ≥ n 时，表示缓冲池已满，生产者等待。
- get(item) 过程：消费者利用该过程从缓冲池中取出一个产品，当 count ≤ 0 时，表示缓冲池已无可取的产品，消费者应等待。
``` c++
Monitor procducer_consumer {
   item buffer[N]; 
   int in out;
   condition notfull, notempty; 
   int count; 
   public: 
   void put(item x) {
      if (count>=N) notfull.wait;       
      buffer[in] = x; 
      in= (in+l) % N; 
      count++;
      notempty.signal;
   }
   void get(item x) {
      if (count<=0) notempty.wait; 
      x = buffer[out]; 
      out = (out+1) % N; 
      count--; 
      notfull.signal;
   }
   { in=0; out=0; count=0; } 
}PC;

void producer()
{item x;
while(TRUE) {
    produce an item in nextp; 
    PC.put(x); 
}}

void consumer()
{ item x; 
   while(TRUE) { 
        PC.get(x); 
       consume the item in nextc;
}}
```
# 七、进程通信

- **定义**：多道程序环境中，多个进程相互协作而进行必要的信息交换称为进程通信。
>其中信号量机制也属于进程通信范畴，但它效率低，又对用户不透明，因而必须设计一种高效使用的通信机制。
## 1、进程通信的类型
### （1）共享存储器系统
相互通信的进程共享某些数据结构或存储区，其中又包括两种：
- 基于共享数据结构的通信方式
- 基于共享存储区的通信方式
### （2）消息传递系统
进程间通信以"消息"为单位，该方法比较通用，其中又可分为：
- 直接通信方式
- 间接通信方式（信箱方式）
### （3）管道通信
所谓管道是指连接一个读进程和一个写进程并实现通信的共享文件，又称为pipe文件，以字符流的方式交换信息。主要解决下面三个问题：
- 互斥、同步、对方是否存在
## 2、消息传递通信的实现方法
### （1）直接通信方式
- 指发送进程利用OS所提供的发送命令，直接把消息发送给目标进程。
- 要求：发送进程和接收进程都以显式方式提供对方的标识符
#### 通信原语：
- `Send(Receiver, message);` 发送一个消息给接收进程
- `Receive(Sender, message);` 接收Sender发来的消息
#### 消息格式
消息可分为：定长格式、非定长格式
一般包括：
- 消息头（发送者、接收者、长度、类型、编号、发送日期和时间）
- 消息正文（实际发送数据）
#### 通信中进程的同步方式
1. 发送进程阻塞、接收进程阻塞
2. 发送进程不阻塞、接收进程阻塞
3. 发送进程、接收进程均不阻塞
#### 通信链路
- **定义**：消息传递系统中的重要途径，两进程在通信前必须首先建立通信链路。
	- **按建立方式**：显式建立链路、隐式建立链路
	- **按连接方法**：点—点连接通信链路、多点连接通信链路
	- **按通信方式**：单向通信链路、双向通信链路
	- **按链路容量**：无容量通信链路（无缓冲区）、有容量通信链路（有缓冲区）
### （2）间接通信方式（信箱方式）
**信箱（数据结构）的结构：**
- 信箱头（标识符、拥有者、口令、信箱空格数）
- 信箱体（多个信箱格）
**信箱的操作：**
1. 信箱的创建和撤消，信箱属性（公用、私用、共享）
2. 消息的发送和接收，必须使用共享信箱，并利用系统提供的下述通信原语进行通信
   - `Send(mailbox, message) 将一个消息发送到指定信箱;`
   - `Receive(mailbox, message) 从指定信箱中接收一个消息;`
**信箱的类型：**
- 私用信箱：进程自己创建信箱并作为进程的一部分，进程自己创建信箱并作为进程的一部分，自身可从中读取消息，其它进程则只能向其发送消息
- 公用信箱：由OS创建，各个进程利用通信原语实施通信
- 共享信箱：由某个进程创建，但需指定信箱的拥有者和共享者
**通信进程之间的关系：**
- 一对一关系（专用通信）
- 多对一关系（又称客户／服务器模式）
- 一对多关系（广播方式）
- 多对多关系（公用信箱）

``` c++
    i.next=0;   
    getid(PCB set, receiver.j);  //获得接收进程内部标识符；
    wait(j.mutex);   
    insert(&j.mq, i);     //将消息缓冲区插入消息队列；   
    signal(j.mutex);   
    signal(j.sm);   
}
```

# 八、线程
## 1、线程的引入
如果说，在OS中引入进程的目的是为了使多个程序能并发执行，以提高资源利用率和系统吞吐量，那么，在操作系统中再引入线程，则是为了减少程序在并发执行时所付出的时空开销，使OS具有更好的并发性。
### （1）进程的两个基本属性
1. 进程是一个可拥有资源的独立单位
2. 进程同时又是一个可独立调度和分派的基本单位
### （2）程序并发执行所需付出的时空开销
为使程序能并发执行，系统必须进行以下的一系列操作：
1. **创建进程**：系统在创建一个进程时，必须为它分配其所必需的、除处理机以外的所有资源
2. **撤消进程**：系统在撤消进程时，又必须先对其所占有的资源执行回收操作
3. **进程切换**：对进程进行上下文切换时，需要保留当前进程的CPU环境，设置新选中进程的CPU环境
### （3）线程 — 作为调度和分派的基本单位
将进程的上述两个属性分开，由OS分开处理，即不把作为调度和分派的基本单位也同时作为拥有资源的单位，以做到"轻装上阵"。
## 2、线程与进程的比较
1. **调度的基本单位**
2. **并发性**
3. **拥有资源**
4. **独立性**
5. **系统开销**
6. **支持多处理机系统**
## 3、线程的状态和线程控制块
### （1）线程运行的三个状态
与传统的进程一样，在各线程之间也存在着共享资源和相互合作的制约关系，致使线程在运行时也具有间断性。
- 所以线程在运行时也具有下述三种基本状态：
`执行状态`
`就绪状态`
`阻塞状态`
### （2）线程控制块（TCB）
系统也为每个线程配置了一个线程控制块TCB，将所有用于控制和管理线程的信息记录在线程控制块中。
### （3）多线程OS中的进程属性
通常在多线程OS中的进程都包含了多个线程，并为它们提供资源。OS支持在一个进程中的多个线程能并发执行，但此时的进程就不再作为一个执行的实体
- 多线程OS中的进程有以下属性：
	1. 进程是一个可拥有资源的基本单位
	2. 多个线程可并发执行
	3. 进程已不是可执行的实体
