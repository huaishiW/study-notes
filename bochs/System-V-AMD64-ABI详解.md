# System V AMD64 ABI 详解

> **System V Application Binary Interface — AMD64 Architecture Processor Supplement**
>
> 这是 Linux（以及 FreeBSD、Solaris、macOS 等 UNIX 系统）在 x86_64 架构上共同遵循的二进制接口标准。所有遵循该 ABI 编译出来的目标文件、共享库和可执行文件，可以跨不同操作系统和不同编译器实现互操作。

---

## 1. ABI 是什么？它解决什么问题？

**ABI（Application Binary Interface，应用二进制接口）** 是比 API（源代码级别）更低一层的约定，规定了：

- **数据表示**：各数据类型的字节大小和对齐方式
- **调用约定**：函数参数如何传递（寄存器 / 栈）、返回值如何处理、哪些寄存器需要调用方/被调用方保存
- **内存布局**：栈帧的布局、函数调用序列的完整细节
- **符号修饰**：函数名和变量名在目标文件中的修饰（name mangling）
- **目标文件格式**：使用的可执行文件格式（ELF）及各节（section）的语义
- **处理器状态**：寄存器用途划分、特殊状态位（MXCSR 等）

ABI 不关心源代码怎么写，只关心**编译后二进制之间的交互**。只要两个编译器都遵循同一个 ABI，它们产生的目标文件可以链接在一起，运行时也可以互相调用。

---

## 2. 数据表示（Data Representation）

### 2.1 基本标量类型

ABI 规定了在 C/C++ 语言层面的类型与 x86_64 硬件表示之间的对应关系。x86_64 使用 **LP64 数据模型**（long 和指针为 64 位，int 为 32 位）:

| C 类型 | 大小（字节）| 对齐（字节）| 说明 |
|--------|------------|------------|------|
| `char` | 1 | 1 | 有符号字节 |
| `short` | 2 | 2 | 有符号 16 位 |
| `int` | 4 | 4 | 有符号 32 位 |
| `long` | 8 | 8 | **LP64 中为 64 位** |
| `long long` | 8 | 8 | 有符号 64 位 |
| `pointer` | 8 | 8 | 64 位虚拟地址 |
| `float` | 4 | 4 | IEEE 754 单精度 |
| `double` | 8 | 8 | IEEE 754 双精度 |
| `__m64` | 8 | 8 | MMX 寄存器值 |
| `__m128` | 16 | 16 | SSE 寄存器值（需对齐） |

> **与 ILP32（32 位）的核心区别**：在 32 位系统上 `long` 是 4 字节；在 LP64 上 `long` 是 8 字节。这是在写跨平台代码时最容易出错的类型之一。

### 2.2 字节序

**小端序（Little-endian）**：最低有效字节存放在最低地址。这是 AMD64 架构强制的，所有符合 AMD64 ABI 的实现必须使用小端。

### 2.3 浮点控制状态（MXCSR）

MXCSR 是 AMD64 SIMD 指令集（SSE/FPU）使用的控制/状态寄存器。ABI 定义了其掩码位的初始值：

```
MXCSR 初始值 = 0x1F80
  Bits 0-5:   异常屏蔽位（全部置 1，即屏蔽所有异常）
  Bits 6:     清零标志（FZ），初始为 0
  Bits 7-15:  保留
  Bits 16-31: 保留
```

程序启动时，动态链接器确保 MXCSR 处于此初始状态，编译器在生成使用 SIMD 指令的代码时也必须遵循此约定。

## 3. 函数调用约定（Function Calling Sequence）

这是 AMD64 ABI 最核心的部分，规定了函数调用时数据的流动方向。

### 3.1 整数寄存器用途划分

AMD64 提供 16 个通用寄存器（GPR），ABI 按用途做了严格划分：

| 寄存器 | 别名 | 保存要求 | 用途 |
|--------|------|---------|------|
| `rax` | — | **调用方保存（caller-saved）** | 返回值（整数/指针）；汇编辅助（临时） |
| `rbx` | — | **被调用方保存（callee-saved）** | 通用寄存器 |
| `rcx` | — | caller-saved | 第 4 个整数参数 |
| `rdx` | — | caller-saved | 第 3 个整数参数 |
| `rsi` | — | caller-saved | 第 2 个整数参数 |
| `rdi` | — | caller-saved | **第 1 个整数参数** |
| `rbp` | — | callee-saved | 帧指针（可选，见栈帧章节） |
| `rsp` | — | 特殊 | 栈指针（不可破坏） |
| `r8` | — | caller-saved | 第 5 个整数参数 |
| `r9` | — | caller-saved | 第 6 个整数参数 |
| `r10` | — | caller-saved | 汇编辅助/特定用途（如 PIC 相关） |
| `r11` | — | caller-saved | 汇编辅助 |
| `r12` | — | callee-saved | 通用寄存器 |
| `r13` | — | callee-saved | 通用寄存器 |
| `r14` | — | callee-saved | 通用寄存器 |
| `r15` | — | callee-saved | 通用寄存器 |

> **callee-saved 寄存器**（rbx, rbp, r12-r15）：被调用函数如果使用这些寄存器，必须先保存它们的值并在返回前恢复。对编译器生成的代码和手写汇编都有约束。
>
> **caller-saved 寄存器**（rax, rcx, rdx, rsi, rdi, r8-r11）：被调用函数可以自由使用，调用方需要在调用前自行保存（如有必要）。

### 3.2 整数参数传递（Register Parameter Passing）

前 **6 个**整数/指针参数通过寄存器传递，**按顺序**使用：

```
第 1 参数 → rdi
第 2 参数 → rsi
第 3 参数 → rdx
第 4 参数 → rcx
第 5 参数 → r8
第 6 参数 → r9
```

超过 6 个的参数，通过**栈传递**（见 3.4 节）。

### 3.3 返回值传递

- **整数/指针返回值**：`rax`（如果返回值大于 8 字节，则使用 `rdx:rax` 联合返回，rax 存低 8 字节）
- **浮点返回值**：`xmm0`（单精度/双精度）；`xmm1:xmm0` 联合返回更宽的向量类型
- **结构体返回值**：如果返回对象较大（大于 16 字节），调用方会在栈上分配一块隐藏空间（hidden storage），并将该空间的地址通过 `rdi` 传递（此时参数序号顺移一位），被调用方将返回值复制到该隐藏空间中。这种情况叫 **RVO（Return Value Optimization）** 但 ABI 也为此留了约定。

### 3.4 栈帧布局（Stack Frame Layout）

以下是符合 AMD64 ABI 的标准栈帧布局（以 `rsp` 递减方向增长）：

```
高地址
┌─────────────────────────┐
│      调用方返回地址       │  ← CALL 指令自动压入
├─────────────────────────┤
│      参数 7+（若有）     │  ← 超过 6 个的参数依次入栈
│          ...            │
│      参数 N（最高地址）   │
├─────────────────────────┤
│   16 字节对齐填充        │  ← 使栈帧 16 字节对齐
├─────────────────────────┤ ← 新 rbp（如果用了帧指针）
│      保存的 rbp          │
├─────────────────────────┤
│  保存的 callee-saved     │  ← rbx, r12-r15（如被使用）
│      寄存器              │
├─────────────────────────┤
│      局部变量            │
├─────────────────────────┤
│      溢出寄存器空间      │  ← 寄存器参数超过可用寄存器时溢出
├─────────────────────────┤ ← rsp 当前指向这里
│      Red Zone           │  ← 128 字节的"红区"（见下）
└─────────────────────────┘
低地址
```

**Red Zone**：在 AMD64 ABI 中，栈指针 `rsp` 之下 128 字节的区域（`rsp-128` 到 `rsp-1`）被称为 **Red Zone**。这是一个信号安全的优化区域： leaf 函数（不再调用其他函数的函数）可以使用这 128 字节而不需要调整 `rsp`，也不需要额外的栈保护。内核和信号处理程序会保留此区域，因此不可依赖它。

**16 字节栈对齐**：函数入口时，`rsp` 必须对齐到 16 字节边界（具体是 `rsp + 8` 相对于 16 的余数为 0，即 `rsp % 16 == 8`，这是因为 `CALL` 压入的返回地址占 8 字节，使被调用函数入口时 `rsp` 天然差 8）。如果调用方在 `CALL` 前 `rsp` 已经是 16 对齐的，则 `CALL` 后进入被调用函数时 `rsp` 处于 8 mod 16 状态，需要在函数序言（prologue）中调整回 16 字节对齐。

### 3.5 函数序言/尾声（Prologue / Epilogue）

典型的符合 AMD64 ABI 的函数序言：

```asm
# 输入：rsp 已处于 8 mod 16 状态（CALL 已压入返回地址）
func:
    push   rbp             # 保存旧帧指针
    mov    rbp, rsp        # 建立新帧指针
    and    rsp, -16        # 16 字节对齐（向下对齐到 16 倍数）
    sub    rsp, N          # 分配局部变量空间（N 为 16 的倍数）
    ; 此时 rsp 已正确对齐
```

尾声：

```asm
    leave                  # 等价于: mov rsp, rbp; pop rbp
    ret                    # 返回调用方
```

### 3.6 变长参数列表（Variable Arguments）

对于可变参数函数（如 `printf`），`al` 寄存器（`rax` 的低 8 位）必须设置为**已向矢量寄存器传递的浮点参数个数**（用于告诉被调用方有多少个浮点参数）。这允许 `printf` 等函数正确解析混合类型的可变参数。

## 4. 寄存器与 DWARF 映射

调试信息格式（DWARF）需要将硬件寄存器编号映射到抽象寄存器名。AMD64 ABI 定义了标准映射：

| DWARF 编号 | 寄存器名 | 说明 |
|-----------|---------|------|
| 0 | `rax` | 返回值/临时 |
| 1 | `rdx` | 第 3 参数/返回值扩展 |
| 2 | `rcx` | 第 4 参数 |
| 3 | `rbx` | callee-saved |
| 4 | `rsi` | 第 2 参数 |
| 5 | `rdi` | 第 1 参数 |
| 6 | `rbp` | 帧指针（callee-saved） |
| 7 | `rsp` | 栈指针（特殊） |
| 8 | `r8` | 第 5 参数 |
| 9 | `r9` | 第 6 参数 |
| 10 | `r10` | caller-saved/ PIC |
| 11 | `r11` | caller-saved |
| 12 | `r12` | callee-saved |
| 13 | `r13` | callee-saved |
| 14 | `r14` | callee-saved |
| 15 | `r15` | callee-saved |
| 16–31 | `xmm0–xmm15` | SSE 矢量寄存器 |

## 5. AMD64 与 ELF 的关系

### 5.1 ELF 对 AMD64 的标识

ELF 文件头中 `e_machine` 字段：

```c
#define EM_X86_64    62   // AMD64 架构的 ELF 机器码
```

### 5.2 ELF 中的 AMD64 相关节

| 节名 | 用途 |
|------|------|
| `.text` | 可执行代码 |
| `.data` / `.bss` | 已初始化/未初始化数据 |
| `.rodata` | 只读数据（如字符串常量） |
| `.got` / `.got.plt` | 全局偏移表（数据引用重定位） |
| `.plt` | 过程链接表（函数调用桩） |
| `.rela.dyn` / `.rela.plt` | 运行时重定位条目 |
| `.dynsym` / `.dynstr` | 动态符号表及字符串表 |
| `.symtab` / `.strtab` | 静态符号表及字符串表 |
| `.dynamic` | 动态链接信息（动态标签数组） |
| `.shstrtab` | 节名字符串表 |
| `.comment` | 编译器版本信息 |

### 5.3 动态标签（Dynamic Tags）

`.dynamic` 节中与 AMD64 紧密相关的标签：

| 标签 | 含义 |
|------|------|
| `DT_SYMTAB` | 动态符号表地址 |
| `DT_STRTAB` | 动态字符串表地址 |
| `DT_RELA` | `.rela.dyn` 地址 |
| `DT_RELASZ` | `.rela.dyn` 总字节数 |
| `DT_RELAENT` | `.rela.dyn` 每个条目大小（24） |
| `DT_JMPREL` | `.rela.plt` 地址 |
| `DT_PLTGOT` | GOT/PLT 入口地址 |
| `DT_BIND_NOW` | 存在则表示 eager binding |
| `DT_TEXTREL` | 存在则表示 .text 节中有需要重定位的引用 |

### 5.4 AMD64 重定位类型

ABI 第 4 章定义了 AMD64 特有的重定位类型。完整的重定位类型列表包括：

| 类型名 | 值 | 计算公式 | 典型用途 |
|--------|---|---------|---------|
| `R_X86_64_NONE` | 0 | — | 无操作 |
| `R_X86_64_64` | 1 | `S + A` | 64 位绝对地址 |
| `R_X86_64_PC32` | 2 | `S + A - P` | 32 位 PC 相对 |
| `R_X86_64_GOT32` | 3 | `G + A` | GOT 条目 32 位相对 |
| `R_X86_64_PLT32` | 4 | `L + A - P` | PLT 条目 32 位 PC 相对 |
| `R_X86_64_COPY` | 5 | — | 数据复制（运行时） |
| `R_X86_64_GLOB_DAT` | 6 | `S` | 全局数据指针 |
| `R_X86_64_JUMP_SLOT` | 7 | `S` | PLT GOT 条目（懒绑定） |
| `R_X86_64_RELATIVE` | 8 | `B + A` | 基址相对（PIE/PIC） |
| `R_X86_64_GOTPCREL` | 9 | `G + GOT + A - P` | PC 相对 GOT 引用 |
| `R_X86_64_32` | 10 | `S + A`（截断为 32 位） | 32 位绝对 |
| `R_X86_64_32S` | 11 | `S + A`（符号扩展为 32 位） | 32 位符号扩展 |
| `R_X86_64_IRELATIVE` | 37 | `B + A`（调用解析器） | IFUNC 间接函数 |

## 6. 与 ELF、动态链接器的关系

AMD64 ABI 是**规则制定者**，ELF 是**承载这些规则的载体**，动态链接器（ld-linux-x86-64.so.2）是**规则的执行者**。三者关系如下：

```
┌─────────────────────────────────────────────────────┐
│           编译器 / 汇编器                             │
│   遵循 AMD64 ABI 生成符合规范的机器码                 │
└────────────────────┬────────────────────────────────┘
                     │ 输出: 目标文件 (.o) / 共享库 (.so)
                     ▼
┌─────────────────────────────────────────────────────┐
│           链接器 (ld)                                │
│   遵循 AMD64 ABI 进行:                               │
│   - 符号解析与绑定                                   │
│   - 重定位表生成                                      │
│   - ELF 节布局                                        │
│   - 产生 .rela.dyn / .rela.plt 重定位条目           │
└────────────────────┬────────────────────────────────┘
                     │ 输出: 动态链接的可执行文件 / 共享库
                     ▼
┌─────────────────────────────────────────────────────┐
│           动态链接器 (ld-linux-x86-64.so.2)          │
│   执行 AMD64 ABI 规定的:                             │
│   - 运行时符号解析（使用 .dynsym / .dynstr）        │
│   - 按 ABI 公式应用 .rela.dyn / .rela.plt 重定位    │
│   - 调用约定检查（寄存器/栈参数传递）                │
│   - MXCSR 状态初始化                                 │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
         ┌──────────────────────┐
         │   运行的进程          │
         │   (CPU 执行符合       │
         │   AMD64 ABI 的指令)   │
         └──────────────────────┘
```

**ABI 是规范**：它规定了"汇编语言应该长什么样"、"寄存器应该怎么用"、"栈帧应该怎么建"。编译器（gcc/clang）生成符合 ABI 的机器码，汇编器（as）遵循 ABI 编码目标文件，链接器（ld）遵循 ABI 合并目标文件并生成符合规范的 ELF。

**ELF 是格式**：它提供了 ABI 条文所依赖的二进制结构——符号表、重定位表、动态段——全部以 ELF 规定好的节和结构体来存储和表达。没有 ELF，ABI 的内容就没有承载介质。

**动态链接器是运行时执行者**：ld-linux-x86-64.so.2 读取 ELF 文件的各个节，按照 AMD64 ABI 中定义的重定位计算公式（`S+A`、`B+A` 等），在运行时修改相应的内存地址。它还负责在程序启动时建立符合 ABI 的初始状态（MXCSR、栈对齐、寄存器初始值等）。

## 7. System V AMD64 ABI 与 Bochs、QEMU 的关系

Bochs 和 QEMU 都是**虚拟化/仿真器**，负责**在一种硬件架构上模拟另一种硬件架构**（或同架构的虚拟化）。它们与 AMD64 ABI 的关系体现在以下几个层面：

### 7.1 QEMU（User-Mode 模拟）

QEMU 在 **user-mode 模拟**模式下（如 `qemu-x86_64 ./prog`），会**以动态二进制翻译器的方式运行**，在非 x86_64 宿主机（如 ARM Linux）上运行 x86_64 Linux 程序。

此时：

- QEMU 必须**理解并处理 AMD64 ELF 文件**：读取程序头、解析 `PT_INTERP`（找到 `/lib64/ld-linux-x86-64.so.2`）、加载 `.dynamic` 段中的标签、按照 AMD64 ABI 的重定位规则进行重定位。
- QEMU 内置了一个**小型的 glibc 运行时模拟库**，或者使用宿主机的 glibc（通过 `syscall` 转换层）。
- 程序中的 `CALL` / `RET` 指令、寄存器使用约定、栈帧布局——全部由 QEMU 的 CPU 模拟单元按照 AMD64 ABI 的规范来执行。
- 如果被模拟程序调用系统调用（`syscall` 指令），QEMU 会捕获并转换为宿主机系统的对应调用。

### 7.2 Bochs（全系统模拟）

Bochs 是一个**全系统 x86 模拟器**（支持 IA-32 / x86-64），它模拟的是整个 PC 环境：CPU、内存、BIOS、IDE 控制器等。

- 当 Bochs 模拟运行一个 **Linux x86_64 操作系统**时，内核的 `PT_INTERP` 段指向的动态链接器（ld-linux-x86-64.so.2）本身也是一段符合 AMD64 ABI 的 ELF 文件。
- Bochs 的 CPU 模拟模块必须**完整实现 AMD64 架构的所有指令和寄存器语义**，包括 `syscall` 指令（根据 AMD64 ABI 从用户态进入内核态）、`iretq` 返回、栈的 push/pop、MXCSR 寄存器的读写等。
- 如果要模拟一个运行 Linux 的 x86_64 虚拟机，Bochs 需要处理所有 ELF 格式的加载和 AMD64 ABI 的执行环境初始化。

### 7.3 三者的层次关系总结

```
┌─────────────────────────────────────────────┐
│  应用程序 (ELF 可执行文件)                   │
│  遵循 AMD64 ABI 调用约定和数据表示           │
├─────────────────────────────────────────────┤
│  动态链接器 (ld-linux-x86-64.so.2)          │
│  遵循 AMD64 ABI 解析 ELF，执行运行时重定位   │
├─────────────────────────────────────────────┤
│  内核 (Linux kernel for x86_64)             │
│  遵循 AMD64 ABI 处理 syscall/返回           │
├─────────────────────────────────────────────┤
│  硬件 (x86_64 物理或虚拟 CPU)               │
│  实现 AMD64 架构的指令集和寄存器模型        │
└─────────────────────────────────────────────┘
                    ▲
                    │ "运行于其上"
                    │
    ┌────────────────┴────────────────┐
    │     QEMU / Bochs 模拟器          │
    │  模拟 x86_64 CPU + 完整 PC 环境  │
    │  处理 ELF 加载，执行 AMD64 指令   │
    │  (QEMU 也可 user-mode 只模拟     │
    │   单一进程的动态链接执行)         │
    └─────────────────────────────────┘
```

**总结**：

- **System V AMD64 ABI** 是**规范**——定义了 x86_64 上所有二进制程序应该如何编写和交互。
- **ELF** 是 ABI 规范所依赖的**文件格式**——所有符号表、重定位表、动态段全部以 ELF 结构存储。
- **ld-linux-x86-64.so.2** 是 ABI 规范在**运行时的主执行者**——按照 ABI 规定的重定位公式和调用约定在进程启动时完成所有链接工作。
- **QEMU** 是一个**模拟器**——它模拟 x86_64 CPU，能够加载和执行符合 AMD64 ABI 的 ELF 文件（无论是作为用户态模拟单个进程，还是全系统模拟运行整个操作系统）。
- **Bochs** 同样是一个**模拟器**——专注于 x86/x86_64 全系统模拟，必须完整实现 AMD64 的 CPU 执行模型和 ABI 规定的所有语义。

从底部向上：硬件/模拟器实现 CPU → 内核/链接器负责 ABI 执行 → 应用程序遵循 ABI 编写 → ELF 作为载体承载所有这些约定。
