# Section Header Table（节头表）
- **定义**：描述ELF文件中各个**节区**元信息的表格
- **作用**：
    - 为编译器、链接器和调试器提供**链接视图**
    - 记录每个节区的名称、类型、大小、偏移量、对齐方式及属性等
    - 支持符号解析、重定位和调试信息获取
- **特点**：
    - 位于ELF文件末尾，通过ELF Header中的`e_shoff`定位
    - 每个表项对应一个节区，大小由`e_shentsize`指定
    - **不是运行必需**，可被`strip`工具移除以减小文件体积
    - 可通过`readelf -S`命令查看
# Program Header Table（程序头表）
- **定义**：描述ELF文件中各个**段**信息的表格
- **作用**：
    - 为操作系统加载器提供**执行视图**
    - 指导操作系统如何将文件映射到内存
    - 定义段的加载地址、大小、权限等关键信息
- **特点**：
    - 通常位于ELF文件头之后，通过ELF Header中的`e_phoff`定位
    - 每个表项描述一个段，大小由`e_phentsize`指定
    - **必须存在于可执行文件中**，是程序加载的必要条件
    - 可通过`readelf -l`命令查看
# Section（节区）
- **定义**：ELF文件中承载特定类型数据的**逻辑单元**
- **常见类型**：
    - **.text**：可执行指令（代码）
    - **.data**：已初始化全局/静态变量
    - **.bss**：未初始化变量（不占文件空间）
    - **.rodata**：只读数据（如字符串常量）
    - **.symtab/.dynsym**：符号表
    - **.strtab/.dynstr**：字符串表
    - **.rela.**：重定位信息
    - **.dynamic**：动态链接元数据
- **特点**：
    - 按功能划分的细粒度数据单元
    - 为链接器提供符号解析和重定位所需信息
    - 通常**不可直接执行**，需要链接器处理
# Segment（段）
- **定义**：由一个或多个节区按**内存属性聚合**而成的执行单元
- **常见类型**：
    - **PT_LOAD**：需加载到内存的段（通常有代码段和数据段）
    - **PT_INTERP**：指定动态链接器路径（如`/lib64/ld-linux-x86-64.so.2`）
    - **PT_DYNAMIC**：包含`.dynamic`节区，供动态链接器使用
    - **PT_NOTE/PT_TLS/PT_GNU_STACK**：辅助信息、TLS模板、栈权限控制
- **特点**：
    - 按内存权限（R/W/X）和加载地址组织
    - 是操作系统加载程序的**基本单位**
    - 每个段有统一的内存权限设置
# Section与Segment的核心区别
## 1. 视角差异
- **Section**：**链接视角**的逻辑单元，面向编译器和链接器
- **Segment**：**执行视角**的内存单元，面向操作系统加载器
## 2. 组织方式
- **Section**：按**功能**划分（代码、数据、符号等）
- **Segment**：按**内存属性**聚合（可执行、可读写等）
## 3. 数量关系
- **一个Segment通常包含多个Section**（相同属性的Section被合并）
- **一个Section只属于一个Segment**
## 4. 实际用途
- **Section**：用于**链接、调试和分析**
- **Segment**：用于**程序加载和执行**
# 为何要将 Section 合并为 Segment
##  1. 减少内存碎片
- **问题**：若每个小Section独立加载，会导致大量内存浪费
- **示例**：`.text`占用4097字节，`.init`占用512字节
    - 不合并：需3个4KB页面（浪费7679字节）
    - 合并后：只需2个4KB页面
## 2. 优化权限管理
- 将相同属性的Section合并为一个Segment
    - 代码段：`.text`、`.init`等 → **只读可执行**
    - 数据段：`.data`、`.bss`等 → **可读写**
- 操作系统可**统一设置Segment权限**，提高安全性
## 3. 提升性能
- 减少TLB（转译后备缓冲器）条目数量
- 降低MMU（内存管理单元）的地址转换开销
- 提高程序加载速度（减少页面映射操作）
# magic（魔数）
## 前四个字节
含义：
- **`7F`**:  一个固定的、不可打印的 ASCII 控制字符（DEL）。
- **`45`**:  对应 ASCII 字符 'E'。
- **`4C`**:  对应 ASCII 字符 'L'。
- **`46`**:  对应 ASCII 字符 'F'。
因此，这个魔数在视觉上可以表示为 `.ELF`，其中第一个字符是 `0x7F`。
## 5 ~ 9 字节
- 第 5 字节：文件类别 Class
	- **含义**: 这指明了该 ELF 文件是 64 位 (`ELF64`) 的。如果是 32 位文件，这个值会是 `01`
- 第 6 字节：数据编码 Data
	- **含义**: 这定义了文件内部数据的字节序。
	    - `01` 代表 **小端序 (Little Endian)**，即低位字节存放在低内存地址。
	    - `02` 代表 **大端序 (Big Endian)**。
- 第 7 字节：ELF 版本  Version
	- **含义**: 这表示 ELF 文件头的版本号。目前这个值固定为 `1` (当前版本)。
- 第 8 字节：操作系统 ABI (OS/ABI)
	**含义**: 这指明了文件面向的操作系统和应用程序二进制接口 (ABI)。`00` 代表标准的 **UNIX System V**，这是最常见的类型
- 第 9 字节：ABI 版本（ABI Version）
	- **含义**: 这是 ABI 的版本号。对于 System V，这个值通常是 `0`。
## 10 ~ 16 字节：
- 只是用于填充
- **值**: `00 00 00 00 00 00 00`
- **含义**: 这 7 个字节是保留的，用于填充，确保 `e_ident` 数组总长度为 16 字节。它们通常被设置为 `0`，有点像身份证 18 位的 X
其实不难发现一个在魔数中就已经包括了，部分下面展示出来的信息
# 展示信息中的两个 Version
- 前一个：ELF 文件版本  第7字节已经展示  是包含在 e_ident 数组中的
- 后一个：文件头版本 是 Elf_Ehdr 结构体中 e_version 字段
>解释：**ELF 文件版本** 和 **文件头版本** 在当前的标准下，其数值通常是相同的（都为 1），但它们在概念上指向的是同一个核心信息：即 ELF 格式本身的版本

- **有两个的原因**：
这主要是出于设计的严谨性和历史原因。`e_ident` 作为文件头的“元信息”，需要包含足够的信息来初步解析文件。而 `e_version` 作为文件头结构体的正式成员，为整个文件提供了一个明确的版本标识。
# Type 的取值与含义
## 1. 核心加载类型
这些类型直接参与进程内存空间的构建，是最基础的部分。
- **NULL**
    - **含义**：**未使用**。
    - **作用**：这是一个占位符。它表示这个程序头表项是无效的，加载器应该忽略它。通常用于填充或预留位置。
- **LOAD**
    - **含义**：**可加载段**。
    - **作用**：这是最重要的类型。它指示操作系统将文件中的某一部分内容（由 `p_offset` 和 `p_filesz` 指定）映射或复制到内存中（由 `p_vaddr` 指定）。
    - **细节**：一个 ELF 文件通常包含多个 `PT_LOAD` 段，例如一个用于代码（只读、可执行），一个用于数据（读写）。如果 `p_memsz`（内存大小）大于 `p_filesz`（文件大小），多出的部分会被清零（通常用于 `.bss` 段）。
## 2. 动态链接相关
这些类型主要用于支持动态链接库（Shared Objects）和动态加载器（如 `ld-linux.so`）。
- **DYNAMIC**
    - **含义**：**动态链接信息**。
    - **作用**：这个段包含了一系列结构体（`ElfN_Dyn`），描述了程序依赖哪些共享库、符号重定位信息、初始化函数地址等。它是动态链接器工作的核心依据。
- **INTERP**
    - **含义**：**程序解释器**。
    - **作用**：这个段包含一个以 null 结尾的字符串，指定了动态链接器的路径（例如 `/lib64/ld-linux-x86-64.so.2`）。
    - **流程**：当内核执行一个动态链接的 ELF 文件时，它会先读取 `PT_INTERP` 指定的解释器，并先运行解释器，由解释器负责加载程序所需的共享库和进行重定位。
## 3. 辅助与元数据
这些类型提供额外的信息，用于调试、异常处理或加载器自身的定位。
- **PHDR**
    - **含义**：**程序头表本身**。
    - **作用**：这个段标记了程序头表（Program Header Table）在文件中的位置和大小。
    - **用途**：动态链接器在运行时可能需要访问程序头表，这个类型帮助它在内存中找到自己的位置。
- **NOTE**
    - **含义**：**辅助信息/备注**。
    - **作用**：包含一些辅助信息，比如构建 ID（Build ID）、核心转储（Core Dump）时的寄存器状态等。这些信息通常不直接参与程序执行，但对调试和追踪很有用。
- **TLS**
    - **含义**：**线程局部存储**。
    - **作用**：定义了线程局部存储（Thread-Local Storage）模板的大小和位置。当创建新线程时，系统会根据这个模板为每个线程分配独立的存储空间。
## 4. GNU/Linux 特有扩展
这些不是 ELF 标准定义的，而是 GNU 项目（GCC/binutils）扩展的，在 Linux 环境下非常常见。
- **GNU_STACK**
    - **含义**：**栈权限指示**。
    - **作用**：它不加载任何数据，而是告诉内核应该以什么权限（读/写/执行）创建进程的栈。
    - **安全**：现代编译器默认生成不可执行的栈（`RW-`），以防止栈溢出攻击。如果没有这个段，内核可能会默认给栈加上可执行权限（`RWE`），这是不安全的。
- **GNU_RELRO**
    - **含义**：**只读重定位**。
    - **作用**：这是一种安全机制（Relocation Read-Only）。它指示链接器将某些数据段（如全局偏移表 GOT 的前半部分）在重定位完成后标记为只读。这可以防止攻击者通过修改 GOT 表来劫持程序流程。

# Align 字段的核心作用
1. **内存权限管理**  
    现代操作系统使用分页机制来管理内存，MMU 的权限保护（如可读、可写、可执行）是以“页”为单位进行的。一个内存页只能设置一种权限。因此，为了保证安全和正确的权限隔离，不同类型的段需要加载到不同的内存页中。
    - **代码段**：通常需要**可读、可执行**（R E），但不能写入，防止代码被篡改。
    - **数据段**：通常需要**可读、可写**（R W），但不能执行，这是一种重要的安全机制（如 W^X 或 DEP），可以防止数据区被当作代码执行。
    `Align` 字段确保了这些段能够被精确地放置在页边界上，从而实现权限的有效隔离。
2. **提升访问效率**  
    CPU 访问对齐的内存地址通常比访问未对齐的地址要快得多。遵循对齐要求可以优化程序的运行性能
- **解读 Align 的值**：通常是 2 的幂次方。最常见的值与系统的内存页大小相关  将显示的16进制数转换为 10进制就是内存页的实际大小
# 关联规则
- 表述：对于一个可加载段（`PT_LOAD`），其虚拟地址（`p_vaddr`）与文件偏移（`p_offset`）在模对齐值（`p_align`）的意义下必须相等
或者更通俗地说：`p_vaddr` 和 `p_offset` 对 `p_align` 取余数的结果必须是一样的。
## 规则的核心目的：高效的内存映射
这个规则的根本目的是为了配合操作系统的**分页（Paging）机制**，实现高效的内存映射（Memory Mapping）。
1. **分页机制**  
    现代操作系统不会以单个字节为单位来管理内存，而是将内存和磁盘文件都划分为固定大小的块，这个块叫做“页”（Page）。在 x86 架构上，最常见的页大小是 4KB（即 `0x1000` 字节）。内存的分配、回收和权限管理都是以“页”为单位的。
2. **内存映射**  
    当加载一个 ELF 文件时，操作系统（通过 `mmap` 等系统调用）并不会把文件内容简单地“复制”到内存里。更高效的做法是，它会在进程的虚拟地址空间中划出一块区域，并建立一种“映射关系”，告诉 CPU：“当你访问这块虚拟地址时，实际去读取磁盘文件上的对应位置”。
## 为什么需要这个规则？
现在，我们把这个规则放到分页和内存映射的场景中来看：
- **`p_align` 与页大小**  
    对于 `PT_LOAD` 段，`p_align` 的值通常是系统页大小的整数倍，最常见的就是等于页大小（如 `0x1000`）。这意味着，这个段在内存中的起始地址必须是页对齐的。
- **映射的便利性**  
    操作系统的加载器在处理一个 `PT_LOAD` 段时，需要完成以下工作：
    1. 在进程的虚拟内存中找到一块足够大的、空闲的、且地址满足 `p_vaddr` 对齐要求的区域。
    2. 将这块虚拟内存区域与 ELF 文件中从 `p_offset` 开始的 `p_filesz` 个字节建立映射。
>如果 `p_vaddr` 和 `p_offset` 都遵循 `p_align`（即页大小）对齐，那么加载器的工作就变得异常简单：它只需要将文件中的第 N 个页，直接映射到内存中的第 M 个页即可。这种“页对页”的映射是硬件 MMU（内存管理单元）原生支持的，效率极高。

**试想一下，如果没有这个规则会怎样？**  
>假设 `p_offset` 是 `0x1001`（没有页对齐），而 `p_vaddr` 是 `0x4000`（页对齐）。当 CPU 访问虚拟地址 `0x4000` 时，MMU 会去查找它映射的文件页。但它会发现，它需要的是文件中 `0x1000` 页的后半部分和 `0x2000` 页的前半部分拼接起来的数据。这会让内存映射变得极其复杂和低效，操作系统可能被迫放弃高效的 `mmap`，转而使用耗时的内存拷贝（`memcpy`）来完成加载。
